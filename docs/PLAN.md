# Codex-Claude Dev Loop 长期开发计划

## 1. 项目定位

Codex-Claude Dev Loop 是一个运行在本地的 AI 编程协作与任务编排工具。

用户在网页端选择本地 Git 项目并输入一次具体开发需求，工具负责组织 Claude 开发、测试执行、Codex 审查、Claude 修复和再次审查的协作流程。

项目的重点不是替代 Git、Claude 或 Codex，而是统一管理：

* 项目与工作区；
* 开发任务；
* 开发计划和提示词；
* Claude实施过程；
* Git修改记录；
* 测试结果；
* Codex审查结果；
* 多轮修复；
* 日志与历史任务；
* 候选版本和安全更新。

## 2. 长期目标

最终希望实现以下流程：

用户输入一次功能需求
→ 工具创建任务
→ 选择安全开发工作区
→ 主 AI 生成本次实施计划
→ Claude执行代码修改
→ 工具运行测试并收集 Git diff
→ Codex进行只读审查
→ Claude根据审查结果修复
→ Codex再次审查
→ 通过后生成候选版本
→ 用户决定提交、合并或放弃

项目未来还需要支持使用稳定版工具开发工具自身，但必须通过独立 Git Worktree 完成，不能直接修改正在运行的稳定版目录。

## 3. PLAN.md 的作用

本文件只是长期项目备忘录，用于记录：

* 项目整体方向；
* 长期架构；
* 不可违反的约束；
* 当前开发阶段；
* 已完成能力；
* 后续路线。

本文件不是一次具体开发任务，也不要求 AI 在一轮开发中实现其中全部内容。

每次实际开发范围必须以任务控制中的：

* 标题；
* 描述；
* 验收标准；

为唯一直接依据。

如果任务要求与本文件的长期安全约束冲突，应优先遵守本文件中的安全约束，并停止执行冲突部分。

## 4. 当前运行方式

当前稳定版工具目录：

E:\AI-Tools\codex-claude-dev-loop

当前自我开发 Worktree：

E:\AI-Tools\codex-claude-dev-loop-dev

稳定主干：

master

开发分支：

feature/self-development-v2

稳定版用于：

* 启动当前可用的网页端和本地服务；
* 管理其他项目；
* 管理开发版 Worktree；
* 在新版本通过验收后合并开发分支。

开发版用于：

* Claude修改代码；
* Codex审查代码；
* 运行单元测试和端到端测试；
* 验证新功能；
* 生成候选更新。

任何 AI 都不得为了开发新功能而直接修改正在运行的稳定版目录。

## 5. 当前阶段

当前处于：

**安全自我开发能力的基础建设阶段。**

当前重点不是自动替换正式版本，而是先做到：

1. 稳定版和开发版目录分离；
2. 工具能够识别并管理 Git Worktree；
3. 同一 Git 仓库的不同 Worktree可以作为独立工作区注册；
4. Claude、测试和 Codex审查只操作开发 Worktree；
5. 开发通过后生成候选更新状态；
6. 用户手动决定是否合并回 master；
7. 开发失败时可以安全放弃 Worktree。

## 6. 已完成或已有基础

项目目前已有以下基础能力：

* 网页端可以调用本地 CLI；
* 可以注册并管理本地项目；
* 已有 Claude开发和 Codex审查的协作流程；
* 可以通过 Git status 和 Git diff 收集修改；
* 已经实践过开发分支合并回主干；
* 已经创建独立 Worktree用于开发工具自身；
* 已开始完善项目移除、归档和生命周期管理。

具体功能是否完整，应以当前代码、测试结果和任务记录为准，不能只根据本文件判断。

## 7. 长期模块规划

### 7.1 项目与工作区管理

需要支持：

* 普通本地项目；
* Git主工作区；
* Git Worktree；
* 同一仓库注册多个不同工作目录；
* 项目归档和恢复；
* 从工具中移除项目；
* 工作区类型和当前分支展示；
* 项目路径白名单；
* 运行中项目保护。

### 7.2 任务管理

每个任务需要保存：

* 标题；
* 描述；
* 验收标准；
* 目标项目；
* 目标工作区；
* 当前阶段；
* 当前审查轮数；
* 最大审查轮数；
* Claude实施报告；
* Git修改；
* 测试结果；
* Codex审查结果；
* 最终状态；
* 完整日志。

### 7.3 AI协作流程

长期支持：

* Claude规划；
* Claude实施；
* 独立测试；
* Codex只读审查；
* Claude修复；
* Codex复查；
* 最大轮数限制；
* BLOCKED、FAILED、STALLED 和 PASS 状态；
* guided模式；
* 以后可选的 CLI自动模式。

### 7.4 自我开发

工具开发自己时：

* 稳定版不得直接被修改；
* 必须创建开发分支；
* 必须使用独立 Worktree；
* AI只能操作 Worktree；
* 测试和审查也必须在 Worktree中执行；
* 通过后生成候选更新；
* 默认不自动合并；
* 默认不自动重启；
* 默认不自动回滚；
* 用户负责最终应用更新。

### 7.5 候选更新与版本管理

以后逐步实现：

* UPDATE_READY 状态；
* 候选版本修改摘要；
* 测试报告；
* Codex最终审查结论；
* 用户手动应用；
* 合并前检查；
* 合并后健康检查；
* 失败回滚；
* 版本标签；
* 历史版本查看。

## 8. 长期安全约束

无论开发哪个功能，都必须遵守以下约束：

1. 不自动执行 git push。
2. 不默认执行 git commit。
3. 不自动合并到 master。
4. 不执行 git reset --hard。
5. 不执行 git clean。
6. 不直接永久删除用户项目。
7. 不删除工具自身稳定目录。
8. 不操作磁盘根目录。
9. 不读取、展示或记录 .env 和密钥内容。
10. 不允许网页端提交任意系统命令。
11. 子进程调用不得通过 shell=True 拼接用户输入。
12. 项目路径必须经过规范化和白名单验证。
13. 测试失败时不能判定任务通过。
14. Codex报告缺失或格式无效时不能判定任务通过。
15. 自动修复必须设置最大轮数。
16. 稳定版运行期间不得直接覆盖稳定版源代码。
17. 所有失败都必须保留日志和中间产物。
18. AI审查通过不等于自动发布，最终操作由用户决定。

## 9. 开发原则

每次只开发一个明确任务。

每个任务必须具有：

* 清晰标题；
* 明确描述；
* 可测试的验收标准。

开发时应：

* 优先复用已有代码；
* 避免一次性重构整个项目；
* 不实现任务范围外的功能；
* 为新增逻辑补充测试；
* 保留兼容性；
* 将安全保护放在便利性之前；
* 使用 demo 项目或 Worktree验证后再用于真实项目。

## 10. 阶段路线

### 阶段一：项目生命周期管理

* 项目归档；
* 项目恢复；
* 从工具中移除；
* 任务归档；
* 操作日志；
* 安全删除准备。

### 阶段二：Worktree识别与管理

* 识别主工作区和 Worktree；
* 同一仓库允许多个工作区；
* 显示工作区类型和分支；
* 关联父项目；
* 防止重复路径；
* 安全移除 Worktree记录。

### 阶段三：安全自我开发

* 从稳定版创建开发 Worktree；
* 自动生成分支名；
* 将任务绑定到开发 Worktree；
* Claude、测试和Codex只操作开发目录；
* 通过后标记 UPDATE_READY；
* 支持放弃候选更新。

### 阶段四：候选更新应用

* 显示完整修改摘要；
* 显示测试结果；
* 显示最终审查结论；
* 用户手动确认应用；
* 合并前安全检查；
* 合并后重新测试。

### 阶段五：可靠性增强

* 子进程中止；
* 超时；
* 任务恢复；
* 日志脱敏；
* 停滞检测；
* 端到端测试；
* Windows中文和空格路径测试。

### 阶段六：可选自动化

在 Claude CLI 和 Codex CLI 可稳定调用后，再考虑：

* automatic模式；
* 自动规划；
* 自动实施；
* 自动只读审查；
* 自动修复循环。

不得通过模拟鼠标点击的方式强行控制 Codex桌面端或 VS Code插件。

## 11. 当前成功标准

当前阶段的成功不要求实现完全自动更新。

当前阶段只需要证明：

1. 稳定版可以继续正常运行；
2. 开发版拥有独立目录和开发分支；
3. 工具可以把开发 Worktree当作独立工作区管理；
4. Claude只修改开发版；
5. Codex只审查开发版；
6. 测试在开发版运行；
7. 审查通过后可以由用户手动合并；
8. 开发失败时可以安全放弃，不影响稳定版。
